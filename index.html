Today: Big-O Notation

Why do we need this?

Consider the following function:


arr = [5, 2, 6, 2, 8, ..., 12]

def total_sum(numbers):
    total = 0
    for number in numbers:
        total += number
    return total
  • How much time does it take to run this function?
  • How does the runtime of this function grow over time?
Big O Notation

Big O Notation is a fundamental tool used to find out the time complexity of algorithms

  • a tool used to describe the time complexity of algorithms
  • provides an upper-bound runtime (or worst-case complexity) of an algorithm
  • analyzes and classifies algorithms depending on their run time or space requirements
https://en.wikipedia.org/wiki/Big_O_notation
Time Complexity
  • refers to the amount of time an algorithm takes to run
  • calculates the time taken to execute each code statement in an algorithm
Sorting Algorithms

Runtime Complexity for:

  • Binary Search ‚Äì O(log n)
  • Linear Search ‚Äì O(n)
  • Heap Sort, Merge Sort - O(n log n)
  • Bubble Sort, Selection Sort, Insertion Sort, Bucket Sort - O(n^c)
  • Exponential algorithms like Tower of Hanoi - O(c^n)
Binary Search

O(log n)

This is because a binary search algorithm divides the data in half each time, making it very efficient for searching through sorted data.

Linear Search

O(n)

A linear search goes through each item in a list sequentially until the item is found or the end of the list is reached.

Heap Sort, Merge Sort

O(n log n)

These sorting algorithms are more efficient than simple comparison-based sorts like bubble sort because they combine elements in a divide-and-conquer approach which reduces the total number of comparisons needed.

Bubble Sort, Selection Sort, Insertion Sort, Bucket Sort

O(n^c)

The exact exponent c depends on the specific details of the algorithm and data. For Bubble, Selection, and Insertion sort, it's typically O(n^2) for average and worst cases, whereas Bucket Sort's complexity can vary depending on how the data is distributed and the number of buckets used.

Exponential algorithms like Tower of Hanoi

O(c^n)

Exponential algorithms have runtimes that grow very rapidly with the input size, making them impractical for large n. In the Tower of Hanoi, with each increase in the number of disks, the number of moves required doubles plus one additional move.

SQL vs NOSQL

Relational databases (RDBMS) like MySQL and NoSQL databases like MongoDB differ fundamentally in their structure, use cases, and approaches to scalability.

Data Structure
  • Relational:
    • store data in tables with rows and columns
    • using predefined schemas with specific data types
    • defines relationships between tables
  • NoSQL Databases:
    • use a more flexible data structure
    • document-oriented database (json-like)
    • no predefined schemas, dynamic
Schemas
  • Relational:
    • require a predefined schemas
    • alterations to schemas require migrations
  • NoSQL Databases:
    • use dynamic or flexible schemas
    • allow documents in the same collection to have different fields
Query Language
  • Relational:
    • Structured Query Language (SQL)
    • standardized across different database systems
  • NoSQL Databases:
    • different language across different types of databases
Scalability
  • Relational:
    • traditionally scale vertically
    • adding more power (CPU, RAM, SSD) to an existing machine
  • NoSQL Databases:
    • designed to scale horizontally
    • using distributed clusters of hardware to manage large volumes of data and traffic
Use Cases
  • Relational:
    • used when data integrity and consistency are critical
    • suited for complex queries and reports
    • if data is primarily structured and unchanging
  • NoSQL Databases:
    • used for performance, scalability, and flexibility
    • ideal for storing unstructured or semi-structured data
    • used in development where the schema may evolve over time
Examples
  • Relational:
    • financial systems
    • e-commerce platforms
    • other transactional systems
  • NoSQL Databases:
    • big data applications
    • content management systems
    • mobile applications
